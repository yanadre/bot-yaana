BUGS
----

B1 [ ] Log timestamps show UTC instead of local time
     All log lines are hours behind actual local time.
     Fix: set TZ env var in compose.override.yaml, or configure logging.Formatter
          with a converter that applies the local timezone.
     File: app/bot/setup.py

B3 [ ] List toggle label is always "done" (✅/☐) regardless of list type
     For non-task/shopping lists the check action has a different meaning:
     "read" for book_list, "watched" for movie_list/series_list, "visited" for
     place_list, etc.  Currently all lists use the same ✅/☐ toggle and the
     "Remove done items" button, which is confusing.
     Fix: add an optional "checked_label" (and matching emoji) to each type
     definition in STRUCTURED_TYPES (e.g. {"checked_label": "watched",
     "checked_emoji": "👁"}); fall back to "done" / ✅ for unknown *_list types.
     Use this in:
       – render_item_line() in structure_types.py
       – _build_toggle_buttons() and _build_action_rows() in lists.py
       – HITL preview in hitl.py
     Files: app/bot/structure_types.py, app/bot/lists.py, app/bot/hitl.py

B2 [ ] Agent does not create a list when asked to — just replies with plain text
     When the user says "create a movie list from my movies", the agent searches
     the vault correctly but then only writes out the results as prose instead of
     calling add_to_vault to actually create a new list document with items[].
     Root cause: the system prompt and tool descriptions only mention task_list and
     shopping_list; the agent has no instruction for "create a *_list type from
     existing docs", and only task_list/shopping_list are recognised by
     _try_extract_list_doc so even if the agent did create e.g. a movie_list the
     UI would not render.
     Fix: extend STRUCTURED_ITEM_TYPES to include movie_list, book_list, series_list
     and any *_list suffix; add explicit create-list-from-docs instructions to the
     system prompt and add_to_vault tool description.
     Files: app/config.py, app/agent/tools.py, app/bot/structure_types.py,
            app/bot/handlers/chat.py, app/bot/hitl.py, app/bot/lists.py

Next up
-------

F0 [ ] Smart retrieval — adaptive filters, hybrid search, dynamic top_k  [complexity: medium-high]
     The current agent uses a fixed top_k and relies on the LLM to choose between
     a plain semantic search and a metadata filter call. This falls short of the
     design goal where retrieval is fully intelligent:
       (a) Metadata filters: the agent should automatically derive Qdrant filter
           dicts from the semantic content of the query (e.g. "unwatched movies"
           → filter item_type=movie AND status!=watched), not only when the user
           explicitly says "filter by …".
       (b) Hybrid search: combine metadata filtering + semantic similarity in one
           query instead of two separate tool calls. Qdrant supports this natively
           via filtered ANN search.
       (c) Dynamic top_k: the agent should reason about how many results the
           question needs. Narrow lookups ("what did I rate Dune?") → k=1–3.
           Open browsing ("what should I watch next?") → k=10–20. Broad
           aggregations ("summarise all my books") → k=all. Add this reasoning
           to the system prompt and/or search tool description.
       (d) Smart metadata at save time: the system prompt should instruct the
           agent to choose metadata fields that will be useful for future
           filtering — e.g. always store "status", "year", "genre" for media
           even if the user didn't mention them, inferring from context where
           possible.
     Files: app/config.py (system prompt), app/agent/tools.py (search tool
            description + schema), app/database/vector_store.py (hybrid search
            support if not already present)

F1 [ ] Due-date reminders  [complexity: medium]
     A daily background job that scans task_list items where due_date <= today
     and sends a Telegram push notification to the user.
     How: use PTB's JobQueue — application.job_queue.run_daily(). Chat ID comes
          from config. Runs inside the existing bot process, no extra service needed.

F2 [ ] Rich item detail / elaboration  [complexity: medium]
     Each list item gets an optional "notes" field for free-form text
     (links, context, sub-steps). Tapping the item opens a detail view as a
     separate bot message.
     How: extend make_item() in structure_types.py; add a
          list_detail_<doc_id>_<idx> callback; render notes in a formatted reply.

F3 [ ] Convenient item editing  [complexity: medium]
     Edit an item's text, priority, effort, or due_date without going through the
     agent. Two design options:
       (a) UI-only: tap ✏️ next to an item → bot asks "send new value"
       (b) Hybrid: shortcut syntax like "new text | high | small | 2026-06-01"

F4 [ ] Delete a single list item  [complexity: small]
     A 🗑 button on each item row. Tap → confirm → item removed and list refreshes.
     How: add list_delete_<doc_id>_<idx> callback + a confirm step, same pattern
          already used for "remove done items" confirmation.

F5 [ ] Agent-driven list editing  [complexity: medium]
     Teach the agent to add, remove, or update individual items in a list via
     tool calls, so the user can say "remove milk from the shopping list" or
     "mark the gym task as high priority".
     How: extend update_vault_metadata tool or add a dedicated patch_list_item
          tool; update the agent system prompt accordingly.

F6 [ ] Streaming LLM responses  [complexity: medium]
     Show a live "⏳ Thinking…" message that updates as the model streams tokens,
     instead of waiting for the full response.
     How: use agent.astream_events(version="v2"), debounce bot.edit_message_text()
          calls every ~300 ms to stay within Telegram rate limits. Fully supported
          by the Telegram Bot API.

F7 [ ] Server deployment  [complexity: small]
     A step-by-step guide for deploying to a production VPS, with a systemd
     service file so the bot auto-restarts on reboot or crash.
     How: SSH + docker compose -f compose.yaml up --build -d + systemd unit file
          with Restart=always and a watchdog ping.
